| TD-001 | Active sidecar watcher: `embed_service` periodically scans the DB for un-embedded rows and embeds them in the background, eliminating the chain race entirely. Today the sidecar is purely reactive — it only embeds when something POSTs to `/embed`. The chain wait-for-socket fix (commit `<TBD>`) closes the race for the wizard's hot path, but a fully self-driving sidecar would make the entire system robust against any timing or watcher failure (e.g. ingest happening while sidecar was briefly down for maintenance). Design sketch: add a background coroutine in `embed_service.py` that wakes every N seconds, runs `SELECT m.rowid, m.content FROM messages m LEFT JOIN message_vecs v ON v.rowid=m.rowid WHERE v.rowid IS NULL LIMIT 100`, and embeds the batch. Needs care around DB lock contention with watcher-driven ingests. | 2026-05-01 | Medium | missing | deliberate | — | Open |

| TD-003 | systemd `.path` units (`PathChanged=` / `PathModified=`) are non-recursive by design — confirmed by `man systemd.path` ("with `PathModified=/var/www/html`, a change in `/var/www/html/dirxx/file.txt` will not trigger the script") and Lennart Poettering's stance ([systemd #4246](https://github.com/systemd/systemd/issues/4246)): "inotify watches are a limited resource [and] this doesn't really scale, and we hence won't do it." Effect: file appends inside `~/.claude/projects/<flat-cwd>/<sid>.jsonl` don't fire the parent-dir watcher we install. macOS launchd's `WatchPaths` IS recursive, so this is Linux-only. **Worked around in Phase 1** by adding response-completion ingest hooks on each CLI (Claude `Stop`, Gemini `AfterAgent`, Codex `Stop`). Phase 2 future: demote systemd `.path` to opt-in. Phase 3 future: replace with `Type=simple recall watch` supervisor. | 2026-05-02 | Medium | structural | auto_confirmed | — | Open |
| TD-004 | `apsw.BusyError: database is locked` raised in `_enable_wal_mode` (`ingest.py:249`) during first `recall install`. **Root cause CONFIRMED experimentally 2026-05-02**: race between two detached subprocesses spawned by the wizard, both calling `_enable_wal_mode`: (a) `recall _backfill-chain` started as `Popen(start_new_session=True)` at `_wizard.py:373`, (b) `recall watch` started per-agent at `_wizard.py:403`. Author's intent (comment at `_wizard.py:329-340`: "Initial ingest must NOT race the watcher's first scan for the WAL writer lock, so the watcher install goes LAST") was correct but implementation broke it — line 373 returns control to the wizard immediately because the backfill child is detached, so the "watcher last" ordering only holds in the parent's main thread, not in actual subprocess startup time. Both children call `con.execute("PRAGMA journal_mode=WAL")` and SQLite needs an exclusive write lock for WAL-mode transition; apsw does not retry on BusyError. Confirmation experiment: ran `recall install --with-embeddings --scheduler polling` interactively answering N to Step 1 (skip watchers) — backfill completed cleanly (66 msgs, 65/66 embedded), zero BusyError. Re-running with watchers=Y reproduces the trace deterministically. Fix candidates: (1) retry-with-backoff in `_enable_wal_mode`; (2) wizard sets WAL mode synchronously before spawning either subprocess; (3) wait for backfill child to clear WAL-init before spawning watcher. See `docs/feedback/2026-05-02-sandbox-phase1.md` Phase I for full experimental log. | 2026-05-02 | High | broken | auto_confirmed | — | Open |
